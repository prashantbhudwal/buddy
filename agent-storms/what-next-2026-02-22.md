## Why Memory Next (vs other paths)

### Memory first is correct now
- It is directly in the product spec and value proposition.
- It leverages infra that already exists (SQLite, multi-tenant scoping, prompt layering, tool permissions).
- It unlocks credible "learning continuity" faster than right-sidebar, MCP, or opencode-runner work.

### Why not right-sidebar first
- Improves presentation, but not learning continuity.
- Risks polishing output for a system that still forgets the learner.

### Why not MCP first
- Valuable, but expands integration surface before core domain loop (memory update/reuse) is stable.

### Why not full parity-sync first
- Full sync is expensive and can become endless.
- Better to do targeted parity sync where it de-risks the memory rollout (prompt/llm/tool loop files).

## Critical Path Analysis

### Critical Path A (must-do pre-step)
1. **Tool contract hardening**
- Add `glob` and `grep` (OpenCode-style) or remove those expectations from prompts.
- Keep `ToolRegistry` and prompt/tool policy aligned.
- Add tests covering invalid tool fallback and prompt/tool compatibility.

Why critical: if the model plans around unavailable tools, memory implementation quality degrades immediately.

### Critical Path B (main build)
1. **Define memory artifact model + file locations**
- Canonical paths under notebook/root:
  - `.buddy/memories.local.md`
  - global memory location (explicitly defined once)
  - optional named memories (`memory.<name>.md`)
- Define merge/update semantics (append-only vs rewrite with compaction rules).

2. **Implement memory read/write services + tools**
- `memory_read`
- `memory_write`
- permission scopes + explicit metadata

3. **Integrate memory into system prompt assembly**
- add memory context layer in `session/system-prompt.ts`
- enforce bounded token budget and compaction strategy

4. **Define write triggers and safeguards**
- post-response update policy
- guard against low-confidence hallucinated facts
- require evidence-driven updates (source message references)

5. **Add minimum UX visibility**
- user can inspect what memory was read/written (transcript/tool output is acceptable in v1)
- do not block on full right-sidebar UX

6. **Quality gates**
- tests for memory persistence across sessions/directories
- tests for permission behavior
- tests for prompt-context loading size bounds

### Dependencies and sequencing
- A1 must finish before B2 to avoid unstable agent behavior.
- B1/B2/B3 are hard dependencies for actual memory continuity.
- B5 can be minimal in v1, but cannot be zero (trust/debuggability risk).

## Risk Analysis

### R1: False or low-quality memories
- Impact: High
- Likelihood: Medium
- Mitigation:
  - write policy requiring confidence threshold
  - include source message IDs in memory entries
  - allow correction/overwrite workflow

### R2: Token bloat from memory injection
- Impact: High
- Likelihood: Medium
- Mitigation:
  - strict token budget for memory context
  - summarize/compact older memories
  - prioritize recency + relevance selection

### R3: Prompt/tool mismatch causing degraded loop behavior
- Impact: High
- Likelihood: High (already observable in config)
- Mitigation:
  - ship `glob`/`grep` or adjust prompt before memory rollout
  - add compatibility tests

### R4: Parity drift silently regressing core runtime
- Impact: Medium-High
- Likelihood: High
- Mitigation:
  - maintain scheduled parity checks
  - targeted sync on P0/P1 files before major memory release

### R5: Over-scoped milestone (memory + sidebar + MCP + multi-provider together)
- Impact: High
- Likelihood: Medium
- Mitigation:
  - keep milestone strict: memory continuity first
  - defer right-sidebar richness and MCP runtime until memory loop is proven

### R6: User trust risk from opaque automatic memory writes
- Impact: High
- Likelihood: Medium
- Mitigation:
  - make memory writes visible in transcript/tool outputs
  - require explicit rationale in write metadata

### R7: Provider lock friction during memory tuning
- Impact: Medium
- Likelihood: Medium
- Mitigation:
  - keep runtime model abstraction clean while shipping with Kimi
  - avoid Kimi-specific assumptions in memory format
