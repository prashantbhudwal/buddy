# Buddy What Next (2026-02-25)

## Recommendation
Build **Buddy Core Logic v1** next.

The vendor architecture cutover is complete enough to start product logic work:
1. Keep OpenCode as runtime authority.
2. Add Buddy-specific learning behavior through adapter seams.
3. Ship learner context + cross-session memory before Tauri/MCP polish work.

If we force one milestone now, it should be:
1. Sync guardrails + regression gates
2. Learner context pipeline (session pulse + history summary)
3. Memory Engine v1 (read/write tools + prompt integration)

## Current State (Code-verified, post-cutover)

### What is strong now
- Single vendor mirror is in place at `vendor/opencode/`.
- Buddy backend is a compatibility facade over vendored OpenCode runtime:
  - `packages/buddy/src/index.ts`
  - `packages/buddy/src/opencode/runtime.ts`
  - `packages/opencode-adapter/src/*`
- Curriculum is wired end-to-end (API, tool registration, UI panel):
  - `packages/buddy/src/opencode/curriculum-tools.ts`
  - `packages/buddy/src/routes/curriculum.ts`
  - `packages/web/src/components/layout/chat-right-sidebar.tsx`
- Compatibility contract surface is still explicit (`/api/*`) and tested.

### What is still missing for core Buddy value
- No learner memory tools yet (`memory_read`, `memory_write`).
- No explicit learner profile/session-history context block in system assembly.
- No quality loop for memory truthfulness and token budget control.
- No tight evaluation harness for "recall across sessions" behavior.

### What changed vs earlier storms
- Earlier "build storage first" guidance is mostly superseded by current runtime posture.
- OpenCode already owns session/message persistence in `opencode.db`.
- Buddy should now focus on behavior-layer differentiation, not rebuilding core loop/storage internals.

## Updated Priority Order

### Priority A: Sync and boundary discipline (must-have guardrail)
- Keep Buddy logic in:
  - `packages/buddy`
  - `packages/opencode-adapter`
- Treat `vendor/opencode/**` as upstream mirror, not feature workspace.
- Make this gate mandatory on vendor sync PRs:
  - `bun run vendor:check-upstream`
  - `bun run typecheck`
  - `bun run test:contracts`
  - `bun run build`

Why first:
- Prevents upstream import churn from silently breaking Buddy behavior work.

### Priority B: Learner Context Pipeline v1
- Add low-cost context that can be computed at request time:
  - session pulse (message count, session age, recent activity shape)
  - lightweight history summary index (last related sessions)
- Inject as bounded context blocks in Buddy-owned prompt assembly path:
  - `buildBuddySystemPrompt()` in `packages/buddy/src/index.ts`

Why now:
- Matches the context-engineering thesis in `agent-storms/learner-context.md`.
- High leverage, low infra cost.

### Priority C: Memory Engine v1
- Implement memory artifacts and service layer in Buddy-owned domain.
- Register tools via OpenCode tool registry pattern (same seam as curriculum tools):
  - `memory_read`
  - `memory_write`
- Keep permissions explicit and auditable (same pattern as curriculum tools).

Why now:
- This is the core differentiator: continuity across sessions.

## Proposed Milestone

### Milestone name
`buddy-core-logic-v1`

### Scope
1. Session pulse context block in prompt injection.
2. Related-session summary index in prompt injection (bounded).
3. Memory artifacts + read/write service.
4. `memory_read` and `memory_write` tool registration through adapter/runtime seam.
5. Contract and behavior tests for cross-session recall.

### Out of scope
- Tauri packaging expansion.
- Full MCP runtime.
- Full multi-provider redesign.
- Large sub-agent catalog expansion beyond what is needed for memory/curriculum loop quality.

### Done criteria
1. A new session can recall relevant facts from prior sessions in same directory scope without re-prompting.
2. Memory reads/writes are visible and inspectable (tool output and/or metadata).
3. Token budget for injected learner context is bounded and tested.
4. Vendor sync gates remain green after implementation.

## Critical Path

1. Define memory data contract and path layout.
2. Build memory service with validation and conflict-safe updates.
3. Register memory tools through OpenCode `ToolRegistry`.
4. Inject learner summary + history index into Buddy prompt merge path.
5. Add eval tests:
   - same-directory recall succeeds
   - cross-directory leakage does not occur
   - stale/low-confidence memory is not promoted without evidence

## Risk Register (for this milestone)

### R1: Upstream internal import churn
- Impact: Medium
- Mitigation: keep all imports behind `@buddy/opencode-adapter`, enforce sync gates.

### R2: Memory quality drift (false memories)
- Impact: High
- Mitigation: evidence-linked writes, confidence tags, correction path, tests for overwrite behavior.

### R3: Token bloat from context injection
- Impact: High
- Mitigation: strict token cap for learner blocks, summary-first strategy, deep fetch only on demand.

### R4: Over-scoped milestone
- Impact: High
- Mitigation: ship context + memory first, defer sidebar richness/MCP/provider expansion.

### R5: Boundary erosion into vendor code
- Impact: Medium
- Mitigation: no feature edits under `vendor/opencode/**` unless intentional patch with explicit tracking.
