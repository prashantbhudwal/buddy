# Codex Prompt Stack (Verified)

## Scope

This note describes how Codex assembles model-visible instructions, verified from source in `~/code/codex/codex-rs/`.

## 1) Base Template Selection

**Source:** `protocol/src/prompts/base_instructions/default.md`

- Codex has a base instruction template with persona, tool behavior, planning, and response formatting guidance.
- The base template is not the only prompt input; runtime additions are appended as separate response items.

## 2) Prompt Assembly Pipeline

**Sources:** `core/src/codex.rs`, `protocol/src/models.rs`

In `build_initial_context(...)`, Codex assembles response items in this sequence:

1. Dynamic permissions (`DeveloperInstructions::from_policy(...)`)
2. Additional developer instructions (if configured)
3. Memory developer instructions (if enabled and available)
4. Collaboration-mode developer instructions (if any)
5. Personality spec message (when personality feature applies)
6. Optional feature sections (apps, commit attribution, etc.)
7. User instructions (`UserInstructions`, includes AGENTS/project docs)
8. Environment context (`EnvironmentContext`)

## 3) Environment Context

**Sources:** `core/src/environment_context.rs`, `core/src/codex.rs`

- Environment is serialized as `<environment_context>...</environment_context>` XML.
- It is emitted as a **user-role message** (`ResponseItem::Message { role: "user" }`), not inlined into the base system string.

## 4) Project Instruction Sources

**Source:** `core/src/project_doc.rs`

- Project docs are AGENTS-first and hierarchical.
- Discovery walks from git root to cwd (inclusive), concatenating docs in order.
- Default filename: `AGENTS.md`.
- Preferred local override: `AGENTS.override.md`.
- Additional fallback filenames can be configured.

## 5) Tools and Permissions

**Source:** `protocol/src/models.rs`

- Sandbox and approval instructions are generated by `DeveloperInstructions::from_policy(...)`.
- Generated permission text includes sandbox mode, approval behavior, and writable roots where applicable.
- Allowed command prefixes are also rendered into dynamic permission instructions.

## 6) Dynamic Runtime Behavior

**Sources:** `core/src/context_manager/updates.rs`, `core/src/memories/prompts.rs`

- Model-switch updates are emitted via `build_model_instructions_update_item(...)`.
- Personality updates are emitted via `build_personality_update_item(...)`.
- Memory read-path instructions are generated by `build_memory_tool_developer_instructions(...)` and include truncated `memory_summary.md`.
- Environment and permission updates can be emitted when turn context changes.

## 7) Side-by-Side Notes vs OpenCode

- Codex emits environment as user-role context; OpenCode injects env in system-side text.
- Codex project-doc ingestion is AGENTS-hierarchy-centric; OpenCode supports AGENTS/CLAUDE/CONTEXT and URL instructions.
- Codex runtime updates (model switch/personality/permissions) are explicit context-manager items.
